# 恋爱先知鸟 (Romantic Oracle Raven) 婚恋匹配应用需求文档

## 1. 项目概述

**致力于用技术打低恋爱所需要的成本。**

恋爱先知鸟（Romantic Oracle Raven）是一个基于主观题问答和向量模型的婚恋匹配应用，通过分析用户的主观回答，生成用户向量模型，在常见婚恋交互场景下提供智能匹配结果。

### 1.1 项目目标
- 提供基于深度学习的智能婚恋匹配服务
- 支持多平台访问（Web、Android、iOS）
- 实现中英双语切换
- 提供场景化匹配功能
- 建立用户向量模型，实现精准匹配

### 1.2 核心价值
- 通过主观题问答深入了解用户价值观和生活方式
- 利用向量模型实现更精准的匹配
- 场景化匹配帮助用户了解特定场景下的匹配度
- 多平台支持提高用户可及性
- 双语支持扩大用户范围

### 2.3 智能匹配系统

#### 2.3.1 匹配算法
- **向量生成**：集成 OpenAI / 通义千问 (Qwen) API，将用户回答转换为 1536 维语义向量。
- **维度分析**：计算相似度（Cosine Similarity）、互补度（Complementarity）和冲突度（Conflict）。
- **综合评分**：Compatibility = (Similarity * 0.4) + (Complementarity * 0.4) - (Conflict * 0.2)。
- **匹配类型**：自动判定为相似型、互补型、合适型或冲突型。
- **[NEW] 聚类预筛选 (Clustering Pre-screening)**：
  - **技术原理**：采用 K-Means 算法将用户分为 5 个特征群体（如“传统家庭型”、“自由探索型”）。
  - **成本优化**：后台预计算群体间的互动结果（Interaction Matrix），避免实时 N² 匹配消耗大量 Token。
  - **AI 预测**：匹配列表页直接调用预计算数据，展示“AI 预测”摘要（无需实时 API 调用）。

#### 2.3.2 匹配结果展示
- 匹配列表展示：以卡片形式展示所有匹配用户，例如每页显示10个匹配卡片
- 匹配分数排序：按匹配分数从高到低排序，例如分数95分的用户排在分数80分的用户前面
- 匹配类型标识：例如相似型显示蓝色标签"相似"，互补型显示绿色标签"互补"，合适型显示紫色标签"合适"，冲突型显示红色标签"冲突"
- 匹配卡片设计：卡片包含用户头像、姓名、年龄、匹配分数和匹配类型，例如"张三 28岁 匹配分数：92 相似型"

#### 2.3.2 匹配详情
- 用户基本信息：显示匹配用户的姓名、年龄、性别、位置、职业等，例如"李四 26岁 女 上海 设计师"
- 匹配分数详情：展示具体的相似度、互补度和冲突度分数，例如"相似度：85%，互补度：70%，冲突度：10%"
- 答案对比：并排显示双方对同一问题的回答，例如左侧显示"你周末喜欢做什么？"的问题，中间显示当前用户的回答"我喜欢看书和健身"，右侧显示匹配用户的回答"我也喜欢看书，偶尔会去爬山"
- 匹配理由说明：系统自动生成匹配理由，例如"你们在价值观上高度相似，在兴趣爱好上有互补性，整体匹配度较高"

#### 2.3.3 场景化匹配
- 场景列表展示：例如包含"共同生活"、"职业发展"、"财务管理"、"旅行规划"等场景
- 场景详情说明：每个场景有详细描述，例如"共同生活场景：评估你们在日常生活习惯、家务分工、生活节奏等方面的匹配度"
- 基于场景的匹配结果：例如在"财务管理"场景下，显示匹配度为85%，提示"你们对财务管理的理念高度一致，适合共同规划未来财务"
- 场景匹配分析：系统生成场景匹配分析报告，例如"在职业发展场景下，你们的职业目标和支持态度互补，有助于共同成长"

### 3.2 架构设计

#### 3.2.1 系统架构
- 前后端分离架构：前端和后端代码分离，通过API通信，例如Web端使用React开发，通过HTTP请求调用后端API获取数据
- RESTful API设计：遵循RESTful设计规范，例如使用GET /api/questions获取问题列表，POST /api/questions/answer提交答案
- 微服务架构预留扩展：系统设计预留微服务拆分能力，例如将来可以将匹配算法拆分为独立微服务
- 容器化部署预留：支持Docker容器化部署，例如使用Dockerfile定义服务镜像，使用Docker Compose管理多容器应用

#### 3.2.2 数据流设计
- 用户注册/登录 → JWT认证：用户注册后，系统生成JWT令牌并返回，例如用户登录成功后，返回{"token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
- 回答问题 → 生成向量 → 存储到数据库：用户提交答案后，系统调用OpenAI API生成向量，例如将用户的所有答案合并后生成1536维向量，存储到用户文档中
- 请求匹配 → 计算匹配度 → 返回匹配结果：用户请求匹配时，系统计算当前用户与其他用户的余弦相似度，例如计算出相似度92%，互补度85%，冲突度10%，返回匹配结果列表
- 场景匹配 → 基于场景计算 → 返回场景匹配结果：用户选择"共同生活"场景时，系统根据该场景的权重计算匹配度，例如返回"在共同生活场景下，你的匹配度为88%"

### 3.3 性能需求

- 页面加载时间 < 2秒：例如Web端首页从请求到完全渲染的时间不超过2秒
- API响应时间 < 500ms：例如调用/api/questions获取所有问题的响应时间不超过500ms
- 支持并发用户数 > 1000：系统能够同时处理1000个以上的用户请求，例如1000个用户同时访问匹配页面
- 数据库查询响应时间 < 200ms：例如查询用户匹配结果的数据库查询时间不超过200ms

### 3.4 安全需求

- 密码加密存储：使用bcrypt等强哈希算法加密存储密码，例如将用户密码"password123"哈希为"$2b$10$EixZaYVK1fsbw1ZfbX3OXePaWxn96p36WQoeG6Lruj3vjPGga31lW"
- JWT令牌认证：JWT令牌设置合理的有效期，例如24小时，并实现刷新机制，例如令牌过期后，用户可以使用刷新令牌获取新的访问令牌
- API请求验证：验证请求头中的Authorization令牌，例如检查令牌是否有效、是否过期，验证请求参数是否符合要求
- 跨域资源共享（CORS）配置：配置允许前端域名访问API，例如允许http://localhost:5173（Web端）和http://localhost:8081（移动端）访问
- 数据传输加密（HTTPS）：生产环境使用HTTPS加密数据传输，例如配置SSL证书，确保所有API请求和响应都通过HTTPS传输

## 5. API设计

### 5.1 认证API

- **POST /api/auth/register** - 用户注册
  - 请求示例：`{"email": "zhangsan@example.com", "password": "password123", "username": "zhangsan"}`
  - 响应示例：`{"success": true, "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}`

- **POST /api/auth/login** - 用户登录
  - 请求示例：`{"email": "zhangsan@example.com", "password": "password123"}`
  - 响应示例：`{"success": true, "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "user": {"id": "666f1f77bcf86cd799439011", "username": "zhangsan", "isPremiumUser": false}}`

- **GET /api/auth/profile** - 获取用户资料
  - 响应示例：`{"id": "666f1f77bcf86cd799439011", "username": "zhangsan", "email": "zhangsan@example.com", "profile": {"name": "张三", "age": 28}, "isPremiumUser": false, "answerCount": 20}`

- **PUT /api/auth/profile** - 更新用户资料
  - 请求示例：`{"profile": {"name": "张三", "age": 29}}`
  - 响应示例：`{"success": true, "message": "Profile updated successfully"}`

- **GET /api/auth/answer-stats** - 获取答题统计信息
  - 响应示例：`{"dailyAnswerCount": 5, "lastAnswerDate": "2026-01-18T00:00:00Z", "isPremiumUser": false, "dailyLimit": 10}`

- **PUT /api/auth/upgrade** - 升级为高级用户
  - 响应示例：`{"success": true, "message": "Successfully upgraded to premium user", "isPremiumUser": true}`

- **PUT /api/auth/downgrade** - 降级为免费用户
  - 响应示例：`{"success": true, "message": "Successfully downgraded to free user", "isPremiumUser": false}`

### 5.2 问答API

- **GET /api/questions** - 获取所有问题
  - 响应示例：`[{"_id": "666f1f77bcf86cd799439012", "text": "你认为什么是幸福？", "textEn": "What do you think happiness is?", "category": "values"}, ...]`

  - 响应示例：`{"_id": "666f1f77bcf86cd799439012", "text": "你认为什么是幸福？", "textEn": "What do you think happiness is?", "category": "values"}`

### 5.3 匹配API

- **GET /api/matches** - 获取匹配列表
  - 响应示例：`[{"_id": "666f1f77bcf86cd799439014", "user2": {"id": "666f1f77bcf86cd799439015", "name": "李四", "age": 26}, "compatibilityScore": 92, "similarityScore": 85, "complementarityScore": 88, "conflictScore": 10, "matchType": "suitable"}, ...]`

- **GET /api/matches/:userId** - 获取匹配详情
  - 请求示例：`GET /api/matches/666f1f77bcf86cd799439015`
  - 响应示例：`{"_id": "666f1f77bcf86cd799439014", "user1": {"id": "666f1f77bcf86cd799439011", "name": "张三", "age": 28}, "user2": {"id": "666f1f77bcf86cd799439015", "name": "李四", "age": 26}, "compatibilityScore": 92, "similarityScore": 85, "complementarityScore": 88, "conflictScore": 10, "matchType": "suitable", "createdAt": "2026-01-18T11:00:00Z"}`

- **POST /api/matches** - 创建匹配记录
  - 请求示例：`{"user1": "666f1f77bcf86cd799439011", "user2": "666f1f77bcf86cd799439015"}`
  - 响应示例：`{"success": true, "match": {"_id": "666f1f77bcf86cd799439014", "compatibilityScore": 92, "matchType": "suitable"}}`

- **POST /api/matches/:userId/scenario** - 生成 AI 场景模拟 (Premium)
  - **功能**：基于双方画像生成特定场景的互动剧本（流式响应）。
  - **请求参数**：
    - `scenarioType`: 场景类型 ('conflict' | 'intimacy' | 'travel' | 'daily_life')
    - `language`: 输出语言 ('zh' | 'en')
  - **响应格式**：Server-Sent Events (SSE)
    - `data: {"content": "..."}`: 增量生成的文本内容
    - `data: {"done": true, "remainingCredits": 19}`: 生成结束及剩余额度

## 8. 测试策略

### 8.1 单元测试
- **后端**: 使用Jest测试核心功能、向量算法、API接口，例如测试userController.js中的register函数，验证用户注册逻辑是否正确
- **Web端**: 使用Vitest和React Testing Library测试组件和功能，例如测试Login组件，验证登录表单提交是否正确触发登录逻辑
- **向量算法**: 测试余弦相似度计算、向量生成逻辑，例如测试vectorService.js中的cosineSimilarity函数，验证计算结果是否准确，输入[0.1, 0.2, 0.3]和[0.1, 0.2, 0.3]应返回相似度1.0
- **API接口**: 测试认证、问答、匹配等API端点，例如测试POST /api/auth/login接口，验证正确的邮箱密码返回JWT令牌，错误的邮箱密码返回401错误

### 8.2 集成测试
- 前后端集成测试：测试前端组件与后端API的交互，例如测试Web端的Login组件与后端POST /api/auth/login接口的集成，验证登录流程是否完整
- 数据库交互测试：测试应用与数据库的交互，例如测试userController.js中的getUserProfile函数，验证是否能正确从MongoDB数据库中获取用户资料
- 第三方API集成测试：测试与OpenAI API的集成，例如测试vectorService.js中的generateVector函数，验证是否能正确调用OpenAI API生成向量，并将向量存储到数据库

### 8.3 端到端测试
- 用户流程测试：测试完整的用户流程，例如从用户注册 → 登录 → 回答问题 → 查看匹配结果 → 查看场景匹配，验证整个流程是否顺畅，数据是否正确传递
- 跨平台测试：测试不同平台（Web、Android、iOS）上的功能是否一致，例如在Web端和移动端都测试登录功能，验证登录流程和结果是否相同
- 多浏览器兼容性测试：测试在不同浏览器上的表现是否一致，例如在Chrome、Firefox、Safari、Edge等主流浏览器上测试匹配结果页面，验证布局和功能是否正常

### 8.4 性能测试
- 负载测试：模拟1000个并发用户访问应用，测试系统在高负载下的表现，例如使用JMeter或LoadRunner模拟1000个用户同时请求匹配结果，验证系统是否能稳定运行，无崩溃或严重性能下降
- 响应时间测试：测试API和页面的响应时间，例如使用Postman测试GET /api/matches接口，验证响应时间是否小于500ms，使用Lighthouse测试Web页面加载时间是否小于2秒
- 资源占用测试：测试服务器在运行时的CPU、内存、磁盘和网络占用情况，例如使用top或htop命令监控服务器CPU使用率，确保在1000并发用户下CPU使用率不超过80%，内存使用率不超过70%

## 9. 部署计划

### 9.1 开发环境
- 本地开发环境：开发人员在本地安装Node.js、MongoDB等依赖，例如在Windows或Mac上使用VS Code开发，本地运行后端服务和Web开发服务器
- Docker容器化开发（支持Docker Compose）：使用Docker Compose定义开发环境，包含MongoDB、Node.js等服务，例如运行`docker-compose up`启动完整开发环境，无需手动安装依赖
- 自动测试集成：在代码提交前自动运行单元测试，例如使用Git Hooks在git commit前运行`npm test`，确保代码质量
- 环境变量配置：使用.env文件管理开发环境配置，例如配置MONGO_URI=mongodb://localhost:27017/datingmatcher-dev，OPENAI_API_KEY=sk-xxx

### 9.2 测试环境
- 独立测试服务器：使用云服务器（如AWS EC2或阿里云ECS）作为测试服务器，例如使用t3.medium实例，运行Ubuntu 20.04系统，与开发环境和生产环境隔离
- Docker容器化部署：使用Docker Compose部署测试环境，包含后端服务、MongoDB数据库和前端应用，例如运行`docker-compose -f docker-compose.test.yml up -d`启动测试环境，确保环境一致性
- 测试数据库：使用独立的MongoDB实例作为测试数据库，与生产数据库隔离，例如使用MongoDB Atlas的M0免费集群，或本地MongoDB实例，数据库名设为datingmatcher-test
- 测试API密钥：使用专门的测试环境API密钥，例如申请OpenAI的测试API密钥，限制调用次数，避免影响生产环境，密钥存储在测试服务器的环境变量中
- 持续集成/持续部署（CI/CD）：使用GitHub Actions或Jenkins实现CI/CD，例如当代码推送到develop分支时，自动构建Docker镜像，部署到测试服务器，并运行集成测试，测试通过后通知开发团队

### 9.3 生产环境
- 云服务器部署：使用云服务器（如AWS EC2或阿里云ECS）部署生产环境，例如使用c5.large实例，运行Ubuntu 22.04系统，配置适当的安全组规则，只开放必要端口
- Docker容器化运行：使用Docker Swarm或Kubernetes管理生产容器，例如使用Kubernetes部署后端服务、前端应用和数据库，实现高可用性和自动恢复
- 数据库集群：使用MongoDB集群（如MongoDB Atlas的M30集群），配置副本集和分片，确保数据高可用和可扩展性，例如配置3个副本节点，分布在不同可用区
- CDN加速：使用CDN（如Cloudflare或阿里云CDN）加速静态资源访问，例如将前端静态文件（JS、CSS、图片）部署到CDN，减少服务器负载，提高访问速度
- 监控系统（Prometheus + Grafana）：部署Prometheus监控服务器资源和应用指标，使用Grafana创建可视化仪表盘，例如监控CPU使用率、内存使用率、API响应时间等，设置告警规则，当CPU使用率超过85%时发送邮件告警
- 日志系统（ELK Stack）：部署Elasticsearch、Logstash、Kibana日志系统，收集和分析应用日志，例如收集后端服务的访问日志和错误日志，使用Kibana查询和可视化日志，便于问题排查
- 负载均衡：使用负载均衡器（如AWS ELB或Nginx）分发用户请求，例如配置负载均衡器将请求转发到多个后端实例，实现负载分担和故障转移
- 自动缩放：配置自动缩放规则，根据CPU使用率或请求数自动调整服务器数量，例如当CPU使用率持续超过70%时，自动添加1个后端实例，当CPU使用率持续低于30%时，自动减少1个后端实例，优化资源使用，降低成本

## 11. 风险评估

### 11.1 技术风险
- 向量模型准确性：例如用户回答的质量参差不齐，可能导致生成的向量不准确，影响匹配结果的可靠性，例如用户随意填写"不知道"或"随便"等答案，导致向量模型无法准确反映用户真实价值观
- 系统性能瓶颈：例如当用户数量达到10万级时，匹配算法的计算复杂度可能导致API响应时间过长，例如计算10万用户之间的余弦相似度需要大量计算资源和时间
- 第三方API依赖：例如OpenAI API可能出现服务中断、价格上涨或访问限制，影响向量生成功能，例如OpenAI API因维护或流量限制无法调用，导致用户无法生成向量，无法获取匹配结果
- 跨平台兼容性：例如在不同浏览器或移动设备上，应用的显示效果和功能可能不一致，例如在IE浏览器上，部分CSS样式无法正常显示，或在某些Android设备上，触摸手势无法正常响应

### 11.2 业务风险
- 用户隐私保护：例如用户的个人信息和回答内容可能涉及敏感信息，需要确保数据安全存储和传输，例如用户的年龄、位置、收入等敏感信息如果泄露，可能导致用户隐私被侵犯，影响用户信任
- 匹配算法接受度：例如用户可能对匹配结果不满意，认为匹配算法不准确，影响用户留存，例如用户回答了大量问题，但匹配结果与用户期望不符，导致用户流失
- 市场竞争：例如市场上已有多个成熟的婚恋交友平台，如Tinder、世纪佳缘等，新平台需要有差异化优势才能吸引用户，例如DatingMatcher的向量匹配算法和场景化匹配是差异化优势，但需要有效推广
- 用户增长挑战：例如新平台初始用户数量少，可能导致匹配结果质量不高，形成恶性循环，例如平台只有100个用户，用户可能很难找到合适的匹配对象，导致用户流失

### 11.3 应对策略
- 持续优化向量模型：例如定期收集用户对匹配结果的反馈，调整向量生成的权重，改进问题设计，例如增加验证问题确保用户认真回答，使用更先进的模型（如GPT-4）生成更准确的向量，提高匹配结果的可靠性
- 性能监控和优化：例如使用Prometheus和Grafana监控API响应时间和服务器资源使用率，设置告警规则，当响应时间超过阈值时自动扩容，优化匹配算法，使用近似最近邻算法（如FAISS）减少计算复杂度，将计算时间从O(n²)降低到O(n log n)
- 备用方案设计：例如准备本地向量生成模型作为备用，当OpenAI API不可用时自动切换，与多个AI服务提供商合作（如Azure OpenAI、Anthropic），实现服务降级机制，确保核心功能可用
- 加强数据安全：例如使用AES-256加密存储用户敏感信息，实施严格的访问控制（最小权限原则），定期进行安全审计和漏洞扫描，使用HTTPS加密所有数据传输，遵守GDPR和CCPA等数据保护法规
- 市场调研和差异化定位：例如定期进行用户调研，了解用户需求和期望，分析竞争对手的优势和劣势，突出DatingMatcher的差异化优势（向量匹配算法、场景化匹配），进行精准营销，针对特定用户群体（如高学历人群）推广应用

### 12.3 安全验收
- 数据加密存储：使用bcrypt等强哈希算法加密存储密码，例如密码"password123"被哈希为"$2b$10$EixZaYVK1fsbw1ZfbX3OXePaWxn96p36WQoeG6Lruj3vjPGga31lW"，无法直接还原，敏感信息使用AES-256加密存储，例如用户的收入、健康状况等敏感信息

- 权限控制有效：实施严格的访问控制，例如普通用户只能访问自己的资料和匹配结果，不能访问其他用户的敏感信息，管理员用户可以访问所有用户数据，但需要额外的认证和授权，例如使用RBAC（基于角色的访问控制）模型
- 漏洞扫描通过：使用OWASP ZAP或Nessus等工具进行漏洞扫描，没有发现高危漏洞，例如没有SQL注入、XSS跨站脚本攻击、CSRF跨站请求伪造等高危漏洞，所有发现的低危漏洞都已修复

## 13. 部署和运行注意事项

### 13.1 MongoDB配置
- **本地MongoDB**: 确保MongoDB服务正在运行，端口为27017
- **Docker部署**: 提供了docker-compose.mongodb.yml文件，可快速启动MongoDB容器
  ```bash
  docker-compose -f docker-compose.mongodb.yml up -d
  ```
- **MongoDB Atlas**: 支持使用云数据库服务，需在.env文件中配置正确的连接字符串
- **连接重试机制**: 后端已实现MongoDB连接重试机制，每5秒重试一次

### 13.2 环境配置
- **端口配置**: 确保端口5000（后端）和27017（MongoDB）未被占用
- **环境变量**: 需配置.env文件，包含以下关键参数：
  - MONGO_URI: MongoDB连接字符串
  - JWT_SECRET: JWT密钥
  - EMAIL_USER: 用于发送密码重置邮件的邮箱
  - EMAIL_PASS: 邮箱密码
  - PORT: 后端服务端口（默认5000）

### 13.3 启动顺序
1. 启动MongoDB服务
2. 启动后端服务器
3. 启动前端开发服务器

### 13.4 常见问题及解决方案
- **MongoDB连接失败**: 检查MongoDB服务是否运行，端口是否正确
- **端口占用**: 使用命令检查并释放占用的端口
  ```bash
  # Windows
  netstat -an | findstr :5000
  # macOS/Linux
  lsof -i :5000
  ```
- **Failed to fetch**: 确保后端服务器正在运行，CORS配置正确
- **数据库缓冲超时**: 检查MongoDB连接状态，确保服务正常

### 13.5 错误处理
- 后端已实现增强的错误处理机制，提供明确的错误信息
- 所有数据库操作都添加了超时处理，防止请求无限等待
- 错误信息会详细说明问题原因，便于调试和修复
